# Canvas 重设计策略稿

## 1. 先纠偏

这次重设计最重要的不是“把 agent 写得更像人”，而是重新回答：

**Canvas 到底是不是主舞台？**

答案应该是：**是，而且必须是。**

参考 OiiOii 的真实工作区体验后，可以明确一件事：

- 用户不是在“跟一个聊天 agent 做项目”
- 用户是在“一个作品工作台上推进创作”
- 对话、agent、参数、生成动作，都只是围绕这个工作台组织

所以这次的北极星不再是“对话驱动创作”，而是：

**让 Canvas 成为真正的创作操作系统。**

## 2. 当前产品的根问题

现状的问题不只是交互不好，而是主舞台错位。

### 2.1 画布不是真正的工作面

现在的画布更像：

- 结果展示区
- 节点陈列区
- 对话之后的回显区

它不是用户真正做决定、做比较、做修改、触发执行的地方。

这会直接导致：

- 用户盯着左侧聊天框做主要判断
- 画布只是“看一眼”
- agent 再强也救不了工作流的空心感

### 2.2 对话抢了主线

现在很多关键决策发生在对话里：

- 方向选择
- 阶段推进
- 触发生成
- 结果解释

这让产品变成了“聊天是主系统，画布是外挂”。

而一旦用户觉得主系统在聊天里，画布就不可能变成真正的创作现场。

### 2.3 agent 分工像演出来的

现在的多 agent 更像阶段切皮：

- 头像变了
- 标签变了
- 但用户没有真正感受到“这是不同工种在围绕同一个作品面协作”

问题不在有没有多个角色，而在：

**这些角色有没有通过画布对象和对象状态体现出真实分工。**

## 3. 从 OiiOii 借什么

不是抄视觉，而是借它成立的产品结构。

### 3.1 借“左侧导演台 + 中央工作面”

OiiOii 的重点不是左边有文案，而是：

- 左侧是导演协作与流程状态
- 中央才是作品对象的真正工作面

这意味着：

- 用户在左侧理解当前任务
- 在中央对对象做操作
- 系统围绕对象反馈，而不是围绕对话回合反馈

### 3.2 借“对象级生成”

OiiOii 不是全局输入一句话然后整页生成。

我实际点进它的画板后，生成输入面板是绑在具体画布对象上的。

这说明它的核心机制是：

- 先有对象
- 再围绕对象配置参数
- 再触发对象级生成
- 再把结果留在对象上下文里

这点我们必须学。

### 3.3 借“空间化选择”

像情绪关键词这种方向选择，在 OiiOii 里不是口头确认，而是视觉卡片选择。

它的价值不是“好看”，而是：

- 决策被空间化了
- 决策能被保留
- 决策能成为后续生成链路的输入

所以我们也不能让候选方案只在对话里短暂出现。

### 3.4 借“后台工种可见，但不多人格抢话”

OiiOii 会让你看到“艺术总监”这类工种存在，但并不会让产品变成多人轮流聊天。

这正是我们需要的：

- 用户感知到真实分工
- 但不会被多个人格切来切去

## 4. 新产品定义

新的定义应该是：

**Canvas = 以作品对象为中心、以对话判断为驱动力的创作工作台。**

其中：

- 画布是主舞台
- 左侧是导演协作台
- 右侧是对象检查与局部制作台
- agent 是围绕对象工作的智能工种系统

这里最关键的补充是：

- **画布负责承载对象、候选、状态、联动**
- **对话负责承载判断、解释、取舍、确认**

不是二选一，而是：

**画布给对话提供“看得见的决策对象”，对话给画布提供“为什么这样选、接下来怎么走”的判断。**

这和“左聊天右画布”的差别非常大。

## 5. 新设计原则

### 原则 1：所有关键动作都要落在对象上

不要再出现：

- 全局聊一轮
- 然后系统神秘地去生成一些东西

必须变成：

- 选中某个对象
- 围绕该对象做判断
- 在该对象上触发生成或修改

对象可以是：

- 方向卡
- 剧本块
- 角色卡
- 场景卡
- 分镜板
- 配音策略卡
- 成片版本卡

### 原则 2：画布上必须同时容纳“候选”和“结果”

现在的问题之一是候选方案没有空间位置。

新的画布必须能同时承载：

- 候选方向
- 当前选中版本
- 已生成结果
- 风险与待确认项

否则用户永远在脑中做比较，而不是在画布上做比较。

### 原则 3：对话只做协作，不做主系统

左侧协作栏负责：

- 说明当前是谁在协作
- 解释为什么这么做
- 给出建议
- 做确认
- 汇报进度
- 帮用户完成取舍
- 把多个对象之间的关系讲清楚

左侧协作栏不负责：

- 代替画布完成选择
- 吞掉所有主线操作
- 成为唯一入口

这条原则不是“削弱对话”，而是给对话明确高价值角色：

- 对话不是操作面
- 对话是决策面

### 原则 4：agent 的价值体现在联动，不体现在说话

真正好的 agent 协作应该是：

- 改一个角色设定，相关镜头提示一起变
- 改一个场景氛围，候选视觉方向跟着刷新
- 锁定一个风格，后续对象继承这套美术基因

不是：

- 多几个身份标签
- 多几段看起来专业的解释

### 原则 5：状态要附着在对象和链路上

状态不能只出现在聊天消息里。

状态必须体现在：

- 某个对象是草案、候选、锁定还是生成中
- 某条链路是待确认、已继承、已失效还是有风险

这样用户才会觉得自己在操作作品，而不是在跟系统猜心思。

## 6. 新信息架构

整个 Canvas 页面建议改成三栏一顶栏，而不是现在这种“对话 + 大片空画布”。

### 6.1 顶部：制作驾驶舱

顶部只放全局信息：

- 项目标题
- 当前创作阶段
- 当前 bottleneck
- 已锁定的核心风格
- 今日产出状态
- 下一步推荐

这是全局导航，不是操作主区。

### 6.2 左侧：导演协作台

左侧不叫聊天框，叫导演协作台。

它负责：

- 当前对接工种
- 当前任务
- 该任务的判断依据
- 推荐动作
- 进度反馈
- 必要时的文字输入
- 当前选中对象的决策说明
- 当前多个候选之间的差异解释
- 当前一步的确认按钮

建议结构：

- 当前协作：编剧 / 艺术总监 / 分镜导演 / 配音导演 / 剪辑顾问
- 当前任务：一句话说明
- 为什么现在做这个
- 当前画布焦点：正在讨论哪个对象
- 你现在可以做的 2-3 个动作
- 文本输入或补充参考入口

导演协作台和普通聊天框的本质区别是：

- 每一轮对话都必须挂靠到一个画布焦点
- 如果没有画布焦点，这轮对话就是低质量对话

### 6.3 中央：作品主舞台

中央是绝对主角。

这里不应该只是节点和线，而应该是：

- 可以比较候选的空间
- 可以放结果的空间
- 可以做局部触发的空间
- 可以看上下游关系的空间

建议中央舞台按“创作对象”组织，而不是按纯流程图组织。

更重要的是，中央舞台要成为“可被对话引用的空间”。

也就是对话里说的每一句关键判断，都应该能对应到：

- 某个对象
- 某组对象
- 某条对象链路

推荐对象类型：

- 创意方向卡
- 剧本结构块
- 角色卡
- 场景卡
- 分镜板
- 声音策略卡
- 版本输出卡
- 风险卡
- 待确认卡

### 6.4 右侧：对象检查台

右侧只服务当前选中的对象。

它负责：

- 对象详情
- 参数设置
- 局部优化
- 相关引用
- 修改影响范围
- 生成按钮

这点很关键：

**生成动作应该主要在右侧对象检查台里发生，而不是在全局消息里发生。**

## 7. 新协作模型

## 7.1 单前台，多工种

对用户来说，前台不需要很多人格。

更合理的结构是：

- 前台身份：导演协作系统
- 当前对接工种：动态切换

也就是说，用户不用和 5 个角色聊天，但必须知道：

- 这轮是谁在负责判断
- 为什么轮到他
- 他对哪个对象负责

### 工种建议

- 编剧：创意方向、结构、对白、角色关系
- 艺术总监：风格、氛围、角色/场景视觉基因
- 分镜导演：镜头组织、节奏、动作与画面叙事
- 配音导演：声音气质、角色区分、旁白策略
- 剪辑顾问：转场、节奏、最终观感
- 制片检查器：判断能否推进、哪里风险最大

## 7.2 工种必须绑定对象责任

不能只显示“当前协作：编剧”。

还必须显示：

- 他正在处理哪个对象
- 处理这个对象的什么问题

示例：

- 当前协作：艺术总监
- 焦点对象：风格方向 A / 分镜板 03
- 本轮任务：锁定整体色温与光感

这样分工才是真实的。

## 7.3 对话与画布必须形成双向驱动

这次真正的重点就在这里。

### 画布 -> 对话

当用户在画布上做动作时，对话必须知道自己该接什么。

典型触发：

- 点击某个对象
- 框选多个候选
- 合并两个方案
- 锁定一个对象
- 标记某个对象有问题

对应的对话行为：

- 解释这个对象现在意味着什么
- 解释多个候选的核心差异
- 说明合并后会带来什么变化
- 在锁定前做最后确认
- 对“有问题”对象给出修法建议

但更重要的一层不是“触发一次回应”，而是：

**对话在后续每一轮里，都必须持续记住当前画布上已经存在什么。**

也就是说，画布到对话不是一次性事件，而是持续上下文注入。

### 对话 -> 画布

当对话里出现关键判断时，画布必须同步变化。

典型触发：

- “这一版更适合走冷感未来风”
- “A 的开场 + B 的结尾更成立”
- “这个角色气质锁定了”
- “可以按这版生成”

对应的画布行为：

- 高亮对应风格对象
- 把两个候选合并生成新对象
- 给角色对象打上已锁定状态
- 在目标对象上出现生成中状态

### 结论

好的交互不是：

- 左边说一通
- 右边自己变

也不是：

- 右边点一下
- 左边跟着废话解释

好的交互应该是：

**用户在任一侧做动作，另一侧都给出最小但关键的反馈。**

更进一步说：

**画布不是对话的外部世界，而是对话的工作记忆。**

## 8. 新主流程

不是简单的 `idea -> script -> storyboard -> voice -> video`。

而是：

`铺对象 -> 比方案 -> 锁对象 -> 触发生成 -> 看联动 -> 再局部修`

阶段仍然存在，但它们通过对象组织，而不是通过聊天轮次组织。

这条主流程还可以拆成两个同步循环：

- **空间循环**：用户在画布上看、选、改、锁
- **决策循环**：用户在对话里听解释、做取舍、给确认

真正成熟的体验，不是让其中一个吞掉另一个，而是让这两个循环咬合。

而这两个循环能否咬合，关键不在“会不会触发”，而在：

- 对话是否持续携带画布上下文
- 画布是否持续承接对话判断

### 8.1 创意期

画布上出现：

- 多个创意方向卡
- 每个方向卡的情绪/风格标签
- 可混搭或合并的候选关系

用户动作：

- 选一个
- 合并两个
- 批注哪一部分喜欢

对话动作：

- 解释几个方向最本质的差别
- 帮用户完成混搭判断
- 在锁定前复述“你真正选中的是什么”

### 8.2 剧本期

画布上出现：

- 多个剧本结构分支
- 每个结构的亮点和风险
- 选中后的主线版本

用户动作：

- 选结构
- 拿 A 的开场 + B 的结尾
- 标记某段要重写

对话动作：

- 说明每个结构的戏剧收益和代价
- 帮用户判断混搭会不会断裂
- 把“重写某段”变成明确修改目标

### 8.3 视觉期

画布上出现：

- 角色对象
- 场景对象
- 风格对象
- 分镜板

用户动作：

- 给角色加参考
- 锁定场景风格
- 点某块分镜局部重做

对话动作：

- 告诉用户该参考会影响哪些对象
- 解释为什么这个风格更适配当前作品
- 给出局部重做的最优路径

### 8.4 声音期

画布上出现：

- 配音策略卡
- 角色声音绑定关系
- 旁白方案

用户动作：

- 选声音方向
- 对个别角色替换
- 决定是否保留旁白

对话动作：

- 解释不同声音策略的情绪结果
- 说明替换个别角色会不会破坏整体统一
- 帮用户判断旁白是不是必要

### 8.5 出片期

画布上出现：

- 版本输出卡
- 当前版本与历史版本
- 风险和建议回看卡

用户动作：

- 预览
- 回滚
- 在某一段重新开分支

对话动作：

- 做回看诊断
- 指出最值得重开的分支
- 帮用户决定是补修还是重出

## 9. 画布交互应该怎么变

### 9.1 从“节点图”变成“对象桌面”

不要再把画布理解成流程连线图。

更适合的心智模型是：

- 导演桌面
- 创作白板
- 作品装配台

对象可以自由摆放、归组、比较、锁定，而不只是按固定列排队。

### 9.2 允许用户在空间里做判断

要支持的核心动作：

- 单击看详情
- 双击进入局部制作
- 框选比较多个候选
- 拖拽重组对象关系
- 合并候选
- 锁定某个版本

### 9.3 让联动可见

当某个对象被修改时，要 visibly 告诉用户：

- 哪些下游对象继承了它
- 哪些对象因此过期
- 哪些对象需要重新确认

这才是 agent 协作最该发力的地方。

### 9.4 让“对话引用对象”成为基础能力

如果我们真的要解决“两边的交互问题”，那就必须让对话能明确引用画布对象。

建议每轮决策都支持：

- 引用一个对象
- 引用多个对象做比较
- 引用一条链路做解释
- 引用一个风险点做确认

也就是说，对话里不能只是说：

- “这个更好”

而应该能明确到：

- “角色卡 A 的气质更适合分镜组 B 的冷感光影，所以我建议锁 A，不锁 C。”

这类引用一旦做出来，用户才会觉得左右两边真的是一个系统。

## 9.5 让“对象状态变化”成为对话的证据

同样地，对话不能只会说结论。

它说的结论，要立刻在画布上留下证据：

- 被选中的对象高亮
- 被放弃的对象降权
- 新合成方案生成新卡
- 已锁定对象出现锁定态
- 待确认对象出现确认标记

只有这样，对话才不是空气。

## 9.6 让“画布状态”成为对话的长期记忆

你刚刚指出的是更本质的一层：

不是“点一下画布，然后对话知道一次”，而是：

**只要画布上已经有这些元素，对话后续就不该忘。**

这意味着对话系统每一轮都不能只带聊天历史，还必须带一份“当前画布状态摘要”。

### 对话每轮至少要知道什么

建议最少要注入四类画布记忆：

1. **当前有哪些对象**
2. **这些对象当前分别是什么状态**
3. **哪些对象已经被锁定/确认**
4. **当前用户焦点停在哪个对象或哪组对象上**

具体到这个产品，至少包括：

- 已存在的方向卡
- 已确认的剧本版本
- 当前角色卡和场景卡
- 已生成的分镜板
- 已锁定的风格与声音策略
- 当前待确认项与风险项

### 为什么这件事重要

如果没有这层“画布记忆”，对话会出现三种典型失忆：

1. **重复提问**
   明明画布上已经锁了一个风格，AI 还在问你想要什么风格。

2. **忽略既有结果**
   明明画布上已经有 3 个候选剧本，对话却像从零开始继续提案。

3. **断开上下文**
   明明用户正在看分镜板 05，对话却在泛泛而谈整个项目。

### 这不是“全量喂给模型”，而是“结构化摘要”

不能把整个画布原样塞给对话。

正确做法应该是每轮生成一份轻量级的 `Canvas Context Summary`，例如：

- 当前阶段
- 当前焦点对象
- 已锁定对象
- 活跃候选对象
- 最近 3 次对象变化
- 当前待确认项
- 当前高风险项

也就是说，对话记住的不是“所有像素”，而是“当前作品状态”。

### 建议的对话记忆分层

建议把对话上下文分成三层：

1. **Conversation Memory**
   聊天历史里已经说过什么

2. **Canvas Memory**
   当前画布上客观存在什么

3. **Focus Memory**
   用户此刻正在看、正在比、正在改什么

只有这三层同时存在，对话才不会失忆。

### 一个简单例子

如果画布上已经有：

- 风格方向 A：未来冷感，已锁定
- 风格方向 B：热烈霓虹，候选中
- 分镜板 01-06：已生成
- 分镜板 03：被用户标记“开场不够抓人”

那么对话下一轮不应该说：

- “你想走什么风格？”

而应该直接说：

- “现在整体已经锁在未来冷感上了，真正悬着的是分镜板 03 的开场抓力。要不要我直接给你两个更强的开场镜头方案？”

这就是“画布记忆进入对话”带来的质变。

## 9.7 画布不是聊天附件，而是系统事实来源

更狠一点说：

在这个产品里，很多时候画布比聊天记录更接近真实状态。

因为聊天记录里可能说过很多探索性判断，但真正已经成立的结果，往往已经落到了画布对象上。

所以设计上应该默认：

- **聊天记录是过程**
- **画布状态是当前事实**

对话系统在继续说话时，必须优先尊重画布事实，而不是只顺着聊天历史往下接。

## 10. 第一版推荐方向

如果基于 OiiOii 的启发，但做我们自己的版本，我推荐方向名叫：

**导演桌面**

它的核心不是“无限画布”本身，而是：

- 用画布承载创意分支和生产对象
- 用左侧协作台承载工种与解释
- 用右侧检查台承载对象级控制
- 用联动系统体现 agent 真正价值

这个方向比“聊天搭档”更适合我们，因为它更能解释：

- 为什么用户要待在 Canvas 里
- 为什么 agent 分工是可信的
- 为什么作品会越做越复杂但不乱

## 11. 这版之后该怎么继续

下一步不该直接画高保真，而应该继续收窄成三件事：

1. 先定义对象体系
2. 再定义三栏架构
3. 再定义工种与对象的责任关系

更具体地说，下一轮最值得直接做的是：

- 输出一版 `Canvas 对象模型`
- 输出一版 `导演协作台信息架构`
- 输出一版 `对象级生成与联动规则`

## 12. 结论

这次真正要重做的，不是“agent 文案”。

真正要重做的是：

- 把画布从结果容器升级成主工作台
- 把对话从主系统调整成决策层
- 把 agent 从会说话的人格改成围绕对象工作的工种系统
- 把生成从全局动作改成对象级动作
- 把创作流从线性流程改成“空间循环 + 决策循环”的双循环协作

如果这五件事做对了，Canvas 才会第一次像一个真正的 AI 创作产品，而不是一个带画布皮肤的聊天流程页。
