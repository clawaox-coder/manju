# Canvas + AI 竞品研究

## 1. 研究目标

这份研究不看“谁也有聊天框”，只看三个问题：

1. 画布是不是主舞台
2. AI 是不是直接作用于对象或选区
3. 对话或 AI 能不能带着工作区上下文继续推进，而不是每轮失忆

## 2. 竞品拆解

## 2.1 OiiOii

最值得关注的点不是“多 agent”，而是：

- 左侧是导演/工种协作栏
- 中央是作品工作面
- 生成动作与参数面板绑定在具体画布对象上
- 候选选择是空间化的，不是纯口头选择

对我们的启发：

- 画布必须是对象工作台，不是结果回显区
- 工种分工必须围绕对象责任来体现
- 候选方案必须能留在画布上而不是闪过

## 2.2 Miro

Miro 的核心思路不是“AI 帮你写几句话”，而是：

- 整个 canvas 本身就是 prompt
- 已有工作内容就是 AI 上下文
- AI 应该嵌进现有工作流，而不是让团队迁就 AI

最值得借的机制：

- 把“已有画布内容”当成上下文资产
- 从 unstructured work 平滑过渡到 structured work
- 把 AI 放进画布上的对象和动作里，而不是独立聊天入口

对我们的启发：

- 不能只把对话历史喂给模型，还要把当前画布事实喂给模型
- 画布对象要能被转成不同结构产物，而不是只停留在同一种形态

## 2.3 FigJam

FigJam 的思路是：

- AI 可以从 prompt 直接生成 board / diagram
- AI 可以直接作用于选中的 stickies
- 输出重新写回 board，而不是停留在对话里

最值得借的机制：

- 选区上下文
- 局部 AI 操作
- 输出回写到同一工作面

对我们的启发：

- 我们的 AI 不能只认“当前阶段”，还要认“当前选中的对象/对象组”
- AI 的结果要落回画布对象，而不是只产出一段文案

## 2.4 tldraw

tldraw 最近最关键的方向是：

- 聊天不够时，直接给 agent 一个完整画布
- agent 可以画，用户也可以改
- agent 能看到用户在画布上的修改

最值得借的机制：

- 画布和对话是真双向同步
- 不是“AI 生成一次图”，而是“AI 和用户在同一画布上持续协作”
- 用户修改不是旁路，而是 AI 的新输入

对我们的启发：

- 用户在画布上的动作必须进入对话记忆
- AI 必须尊重画布当前状态，而不是只顺着文字历史继续说

## 2.5 Whimsical

Whimsical 的强项不在深协作，而在：

- 低门槛把 prompt 直接变成视觉结构
- 从 blank canvas 到可编辑 board 非常快

最值得借的机制：

- 首次起步要足够轻
- 让用户很快从“空白”进入“可编辑对象”

对我们的启发：

- 我们不能让用户先经历太多问答才能看到画布内容
- 第一屏最好很快出现可操作对象

## 3. 共同规律

这些产品虽然路径不同，但有几个共同点非常明显。

## 3.1 画布不是附件

成熟产品几乎都把画布当成：

- 主工作区
- 共享上下文
- 决策与产出发生的地方

不是聊天生成后的截图展示区。

## 3.2 AI 最有价值时，是作用于“已有上下文”

真正强的体验不是“从零 prompt 一次”，而是：

- 基于整个 canvas
- 基于当前选区
- 基于当前对象链路

继续做事。

## 3.3 输出必须回写到工作面

如果 AI 的输出只停在对话里，价值会迅速下降。

成熟产品普遍会把 AI 输出重新落回：

- board
- diagram
- sticky
- object
- visual structure

## 3.4 对话只是其中一种协作层

没有哪家成熟产品把聊天框当唯一主舞台。

真正重要的是：

- 上下文怎么进 AI
- AI 输出怎么回到工作面
- 用户怎么在工作面继续修改

## 4. 对我们的直接结论

## 4.1 该借的

- `OiiOii` 的对象级生成和工种工作台
- `Miro` 的 whole-canvas-as-context
- `FigJam` 的 selected-context AI
- `tldraw` 的双向画布记忆
- `Whimsical` 的低门槛起步速度

## 4.2 不该照抄的

- 不要只抄“多个 agent 标签”
- 不要只抄“左侧一个聊天框”
- 不要只抄“无限画布视觉”

这些都只是表象。

真正该做的是把三件事打通：

1. 画布对象
2. 对话判断
3. 记忆注入

## 4.3 我们最该补的短板

按优先级看，当前最缺的是：

### 第一优先级

对话每轮都带着画布事实继续说。

### 第二优先级

所有生成动作改成对象级，而不是全局阶段级。

### 第三优先级

候选方案能够在画布上并排比较、锁定、合并，而不是只在消息里存在。

## 5. 对后续设计的要求

如果我们继续往下细化，后面所有设计都要满足下面三条：

1. 任何一轮对话，都能指出它正在讨论哪个画布对象
2. 任何一次生成，都能指出它挂靠在哪个画布对象上
3. 任何一个对象变化，都能进入下一轮对话记忆

如果做不到这三条，说明系统还没有真正变成 canvas-first。
