# Canvas 重设计 PRD

## 1. 项目目标

把当前的 Canvas 从“带画布皮肤的聊天流程页”重构成：

**以作品对象为中心、以对话判断为驱动力、以对象级生成和联动为核心的创作工作台。**

这次改版要解决的不是单点体验问题，而是三层系统错位：

- 主舞台错位：画布没有成为真正工作面
- 记忆错位：对话不知道当前画布事实
- 协作错位：agent 分工没有通过对象和联动体现出来

## 2. 成功标准

如果这次改版成功，用户应该明显感受到：

1. 画布是主工作区，而不是回显区
2. 对话不会对已存在的对象和已锁定的事实失忆
3. 任何一次生成都能明确指向某个对象
4. 修改上游对象后，下游对象会产生可理解的联动
5. 多 agent 不再像换皮，而像围绕作品对象工作的工种系统

## 3. 非目标

这次不追求：

- 一次性做完所有对象类型
- 一次性做完所有多人协作能力
- 一次性做完最完整的无限画布体验
- 一次性重写所有后端生成链路

这次优先保证：

- 产品心智模型正确
- 画布/对话/对象三者真正打通
- 第一版核心链路能跑通

## 4. 用户问题

当前用户在 Canvas 上最容易出现这些困惑：

- 不知道主线到底在聊天里还是在画布里
- 对话经常像忘了画布已有内容
- 点画布对象和聊对话像两套系统
- 生成动作像黑箱，不知道是对哪个对象生效
- 改了某个上游方向，下游为什么要不要重做不清楚

## 5. 改版核心方案

## 5.1 页面结构

页面改为：

- 顶栏：制作驾驶舱
- 左栏：导演协作台
- 中栏：作品主舞台
- 右栏：对象检查台

## 5.2 系统核心机制

第一版必须成立的机制：

1. 对象模型
2. Canvas Context Summary
3. 对话输入契约
4. 状态流转与联动规则
5. 对象级生成动作

## 5.3 第一版对象范围

第一版只强保这批对象：

- `idea_direction`
- `style_direction`
- `script_version`
- `storyboard_card`
- `voice_strategy`
- `output_version`
- `annotation`
- `decision_gate`
- `risk_flag`

## 6. 用户主链路

## 6.1 创意期

用户在画布上看到多个方向卡。

左栏解释差异，右栏展示对象详情，用户在画布上选择或合并方向。

输出结果：

- 至少一个 `locked` 的方向对象

## 6.2 剧本期

用户在画布上看到多个剧本版本或结构块。

左栏解释结构收益，右栏支持局部重写，用户锁定一个主剧本版本。

输出结果：

- 一个 `locked` 的 `script_version`

## 6.3 视觉期

用户围绕角色、场景和分镜板工作。

对象级生成从右栏发起，对话带着当前对象记忆继续推进。

输出结果：

- 一批 `ready` 的 `storyboard_card`

## 6.4 声音期

用户锁定声音策略，并可对局部角色覆盖。

输出结果：

- 一个 `locked` 的 `voice_strategy`

## 6.5 出片期

用户看到输出版本与风险对象，并围绕版本回看继续优化。

输出结果：

- 一个或多个 `output_version`

## 7. 第一版必须实现的体验

## 7.1 画布发起 -> 对话承接

当用户点中一个对象时：

- 左栏必须围绕这个对象说话
- 右栏必须展示这个对象详情
- 对话下一轮必须带着这个焦点记忆继续说

## 7.2 对话发起 -> 画布响应

当用户在左栏输入修改意图时：

- 系统必须映射到具体对象
- 画布必须高亮相关对象
- 如果触发决策，要生成对应 `decision_gate`

## 7.3 对象变化 -> 联动解释

当上游对象变化时：

- 相关下游对象进入 `warning` 或 `stale`
- 生成对应 `risk_flag`
- 左栏解释为什么受影响

## 8. 技术设计收敛点

## 8.1 前端

前端需要完成的核心改造：

- Canvas 页布局改造为三栏
- 画布对象体系接入
- 左栏从消息流水改为导演协作台
- 右栏增加对象检查与对象级动作
- 维护 `Focus Memory`
- 生成 `Canvas Context Summary`

## 8.2 后端

后端需要完成的核心改造：

- `/chat` context 升级
- 消费 `canvas_context_summary`
- 消费 `focus_memory`
- 消费 `turn_context`
- 输出更对象化的决策响应

## 8.3 状态与数据层

数据层需要完成的核心改造：

- 统一对象状态模型
- 建立对象关系与联动判断
- 支持 `warning / stale / locked / generating / ready`

## 9. 版本策略

建议拆成 3 个版本推进。

## V1：骨架打通

目标：

- 三栏结构成型
- 画布对象焦点和对话焦点打通
- `/chat` 能接收 `focus_memory + canvas_context_summary`

V1 做完后，至少能让用户感觉：

- 点对象后 AI 不会失忆

## V2：对象级生成

目标：

- 右栏挂对象级生成动作
- `script_version -> storyboard_card`
- `voice_strategy -> output_version`

V2 做完后，至少能让用户感觉：

- 生成不再是黑箱

## V3：状态与联动

目标：

- `warning / stale / risk_flag / decision_gate`
- 上游变化驱动下游状态变化

V3 做完后，至少能让用户感觉：

- 系统真的理解作品因果关系

## 10. 里程碑拆分

建议按下面顺序开工。

### M1 页面壳层

- 三栏布局
- 顶栏驾驶舱
- 左栏导演协作台占位
- 右栏对象检查台占位

### M2 对象接入

- 定义第一版核心对象
- 中栏支持单击焦点与多选比较
- 右栏展示对象详情

### M3 记忆接入

- 前端构造 `Canvas Context Summary`
- 前端构造 `Focus Memory`
- `/chat` 契约升级

### M4 对话重构

- 对话优先围绕焦点对象回应
- 决策动作围绕对象生成
- 避免重复问已锁定事实

### M5 生成重构

- 对象级生成按钮进入右栏
- 生成进度映射到对象状态

### M6 联动规则

- 上游变化 -> 下游 `warning/stale`
- 自动生成 `risk_flag/decision_gate`

## 11. 风险

这次改版主要风险有 4 个。

### 风险 1：对象定义过大

如果第一版对象太多，系统会过重。

应对：

- 先只做核心 9 类对象

### 风险 2：上下文注入过重

如果把完整画布都塞给模型，成本和稳定性都会出问题。

应对：

- 只注入结构化摘要

### 风险 3：前端状态与后端状态不一致

如果对象状态只在前端算，后端理解会跑偏。

应对：

- 明确哪些状态是前端显示态，哪些是产品事实态

### 风险 4：右栏沦为新表单

如果右栏只是堆按钮和参数，会重新变成另一种流程机。

应对：

- 所有右栏操作必须与当前焦点对象和左栏判断联动

## 12. 验收标准

第一版上线前至少验证下面这些场景：

1. 用户点击一个分镜卡，对话能围绕它继续说，不失忆
2. 用户锁定一个风格方向后，对话不再重复问风格
3. 用户修改上游剧本版本后，相关分镜进入 `warning` 或 `stale`
4. 用户在右栏发起对象级生成时，画布对象进入 `generating`
5. 生成完成后，对象状态变为 `ready`，左栏给出对应解释

## 13. 结论

这次改版的本质不是“把 UI 重新排一下”。

它真正是在把当前产品从：

- `聊天主导 + 画布回显`

重构成：

- `对象主导 + 对话决策 + 状态联动`

只要按这个 PRD 顺序推进，后续无论是继续做高保真、拆开发任务，还是直接开始改代码，都会顺很多。
